ADAM Protocol Architecture

A deterministic, Wasm-sandboxed policy engine for on-chain governance. All proposal lifecycles, parameter diffs, and RWA updates are gated by explicit guards, proofs, and bytecode allowlists. Every path is auditable, fuel/memory bounded, and replay-safe.

Constitutional Program

Wasm, deterministic, fixed ABI, fuel+memory capped, no syscalls/time/random. Upgrades via governance only.

Policy Chain

Ordered list per topic. Each returns ALLOW, DENY, AMEND, or REQUIRE_2FA. First terminal verdict wins.

ProofBundle

Deterministic CBOR: EAS UIDs, oracle sigs, countersigns, payload hash, snapshot block. Host keccak256 over CBOR = proofHash.

Proposal Lifecycle

  1. Draft → Submit → Review (≤24h)
  2. Voting (snapshot, tally at end)
  3. Timelock → Challenge Window
  4. Execute (if passed, after challenge)
  5. Monitor (on-chain events, audits)

Static checks at review (targets, bounds, diffs). Challenge uses bonded dispute; ADAM Host re-checks on queue/execute. Emergency: pause/cancel only, dual-quorum enforced.

All parameters are on-chain, versioned, and emitted as JSON. Quorums, supermajorities, proposer bonds, concurrency, and challenge windows are strictly enforced by contract.

Wasm Sandbox

All policy logic runs in a Wasm sandbox (wasm32-unknown-unknown), with strict fuel and memory caps. No syscalls, time, or randomness. Upgrades are only possible via governance.

Host ↔ CP ABI

Host → CP (read-only): getParam, getTopicCfg, getSnapshot, getQuorumReached, getSupermajorityReached, getOracleImpact, verifyEAS, getProofHash, getBlockTime

CP → Host (per hook): onSubmit, onVoteStart, onTally, onQueue, onExecute, onRwaUpdate, onEmergency

Security & Invariants

  • No fund movement without topic TREASURY
  • PARAMS must stay within ParamBounds; monotonic flags enforced
  • RWA updates require (≥2-of-N) ∧ active-operator-stakes ∧ recency window met
  • Emergency: pause() and cancel() only
  • Inter-block 2FA: second, disjoint confirmation within window by SBT-only quorum
  • All paths auditable, gas/DoS bounded, replay-safe

Introducing ARCxSBT

ARCxSBT is a non-transferable, EAS-gated Soulbound Token that proves your role and eligibility in the ARCx ecosystem. Each SBT is:

  • Issued only to eligible contributors (dev, validator, gov, oracle, curator, auditor)
  • Decay-weighted: your on-chain activity keeps your power alive
  • Revocable, with strict anti-DoS and rate limits
  • ERC-721 compatible, but always locked (ERC-5192)
  • Enforces role-level bounds and eligibility for governance
